## Notes

- The meeting is being recorded.  Our code of conduct applies to this meeting.  https://github.com/Ecma-TC54/tg2/blob/main/CODE_OF_CONDUCT.md
- Intro by Philippe, preliminary matters, proposed agenda.
- Minutes from last meeting: approved.
- See the PURL Spec Issues tab for links to popular current issues
- Summary
  - Participants evaluated methods for managing breaking changes in PURL types. They explored versioning individual types versus global releases and integrating versioning into identifiers.
  - The group agreed to simplify the VERS structure by replacing "version-scheme" with "type". This change improves linguistic clarity for component identification and collation definitions.
  - Members decided to establish an outbound newsletter service to notify implementers of official changes. This pilot will launch with the VERS community to streamline communication.
- **Meeting Kick-off and Notes Approval**: The ECMA TC54-TG2 meeting commenced with a reminder of the code of conduct. The meeting notes from the previous session were approved without objection, with participants reminded that any subsequent corrections should be submitted via a pull request online.
- **Agenda Item Collection**: Michael Herzog had added two topics: an ISO update and some VERS items, plus the PURL type for Apache. Philippe Ombredanne noted that discussion on PURL types for Nix and Conda, which are currently the two large ecosystems needing attention, could also be included.
- **Nix Package Type Complexity**: Philippe Ombredanne highlighted the complexity of the Nix package type, noting that packages are often identified primarily by a checksum rather than a simple package name. They explained the potential need to track two PURLs for each Nix package: the upstream PURL, often a 'seed' PURL, and the PURL representing the actual packaging within Nix. Michael Herzog agreed this dual PURL tracking would be a common case for any repackaging of upstream code.
- **Tracking Seed PURLs for Ecosystems**: Philippe Ombredanne stated that tracking a seed PURL is desirable for most Linux distribution packages, as they are frequently repackaged downstream. This issue of tracking different versions of closely related code is a shared concern, especially in C/C++ communities, where different build flags can generate distinct packages, often resulting in new package names in traditional distribution packages.
- **PURL Type Definition Versioning Discussion**: Michael Herzog introduced the topic of versioning PURL type definitions, a question raised by the proposed Conda type changes that necessitate a required namespace. They proposed using a major version change (e.g., 1.0) for "normative" or breaking changes, and minor versions for "editorial" changes like updated examples.
- **Technical Barrier to PURL Type Versioning**: A technical problem was noted regarding the PURL type definition regular expression (regex), which does not currently allow for a version number within the name of the definition. Michael Herzog suggested that for simpler PURL types like SID, starting with a lower version number (e.g., 0.1 or 0.2) might encourage early adoption while allowing for necessary adjustments.
- **Feasibility of Individual Type Versioning**: Philippe Ombredanne argued that managing versioning across potentially 50 to 100 individual types would be impractical due to the resulting "explosion of complexity". Instead, they suggested tracking the history of types in Git and potentially having a single version for the entire set of types.
- **Communicating Breaking Changes and Test Cases**: Michael Herzog emphasized the importance of alerting people and implementation tools to "normative" or breaking changes. They stressed that coordination with test cases is critical, as test cases serve as the "ultimate, detailed definition of [the] spec". They are concerned about how to communicate the version a test case is meant for at a granular level.
- **Impact of Conda Type Breaking Change**: The discussion referenced the pending Conda PURL definition change, which requires a new namespace (the channel), necessitating a breaking change. Michael Herzog noted that the change comes directly from key members of the Conda community who identified the necessity for the channel to work with the Conda ecosystem.
- **Addressing Major Changes by Creating New Types**: Philippe Ombredanne suggested that a better outcome for dealing with major, breaking changes, such as those anticipated for Golang, might be creating an entirely new type definition. Michael Herzog countered that they should not pretend changes will not happen over time and that the issue should be handled by having appropriate test cases for each version.
- **Versioning for Clean Distinction of Type Changes**: Michael Herzog reiterated that versioning types is attractive because it allows for clean distinction of specific normative changes, making communication simpler than relying on change logs and calendar versions (CalVer). Piotr P. Karwasz proposed embedding the version information within the type identifier, such as using `golang+v2`.
- **Preferred Approach to Versioning**: Philippe Ombredanne preferred avoiding version information in the PURL identifier if possible, suggesting having a version for the *whole* set of types, released periodically (e.g., quarterly or biannually). They noted that this approach would make things easier for users and put the burden of tracking changes on implementers.
- **ISO Standardization Update**: Michael Herzog provided an update on the ISO JTC fast-track process for PURL standardization, noting that it is not happening quickly. They received a lengthy document to complete for the process, which is currently "opaque". They have encouraged the SPDX community to update their tools based on the ECMA standard, given the current delay in the ISO process.
- **VERS Naming Convention Proposal**: Michael Herzog proposed changes to the VERS structure, specifically focusing on Issue 66 (https://github.com/package-url/vers-spec/issues/66). They suggested removing the redundant 'version' prefix from component names, changing the first component from `version-scheme` to simply `type`, and changing `version-constraints` to `constraints`. Philippe Ombredanne agreed to use `type` for the first component as it is linguistically simpler.
- **Defining the VERS "Type"**: The participants discussed terminology for the VERS first component, noting that while "scheme" is technically correct, "type" is simpler. Piotr P. Karwasz suggested "collation," as the type often defines how versions are compared (e.g., Maven collation). Philippe Ombredanne agreed that the VERS type identifies the collation (the method for comparing versions).
- **Communicating with Implementers**: Michael Herzog raised the issue of lacking a clean, formal way to communicate official changes to implementers. They proposed using a mailing list, with Groups.io being the most widely adopted platform.
- **Outbound-Only Newsletter Strategy**: Philippe Ombredanne and Michael Herzog agreed that an "outbound only" service, functioning as a newsletter rather than a discussion mailing list, would be the best strategy. This approach avoids issues like spam and encourages discussions to happen on GitHub or other existing channels. They decided to pilot the newsletter with the VERS community, which has a smaller number of initial tool implementers.
- **Next Steps for SID PURL Type**: Philippe Ombredanne committed to creating a pull request for the SID PURL type over the weekend, hoping for a swift approval to begin its implementation. The adoption of the SID type for Apache projects will use the `apache.org` domain, eliminating any ambiguity.
- The meeting was adjourned.
