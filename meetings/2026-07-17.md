## Notes

- The meeting is being recorded.  Our code of conduct applies to this meeting.  https://github.com/Ecma-TC54/tg2/blob/main/CODE_OF_CONDUCT.md.
- Intro by Philippe, preliminary matters, proposed agenda.
- Minutes from last meeting: approved.
- See the PURL Spec Issues tab for links to current Strategic and Tactical issues.
- Summary
  - PURL test schema v0.2 changes are approved.
  - Repository tags will be utilized to archive `purl-test.schema-0.1.json` data rather than maintaining parallel folders. Migration to `purl-test.schema-0.2.json` will occur for the entire repository.
  - Discussion on a single test per file structure highlighted potential improvements for test identification and system uniqueness. This model prioritizes the needs of library implementers over test maintainers.
  - Categorization involves grouping tests by status and functionality through a unified folder structure. Future utility scripts will facilitate test aggregation and failure review.
- **Meeting Opening and Administration**: Philippe Ombredanne called the meeting to order, reminding all participants to follow the TC54 code of conduct, and confirmed that the running notes from the previous meeting were available and approved. Michael Herzog confirmed that four items had been added to the agenda.
- **PURL Test Schema Feedback**: Michael Herzog reported that they had followed the lazy consensus model for proposed changes to the PURL test schema and received positive feedback from Jan Kowalleck and Giuseppe Di Terlizzi, which they interpreted as approval to proceed with publishing the new test schema.
- **Test Schema Versioning Strategy**: Michael Herzog proposed creating a subfolder to version the tests, allowing them to keep the version 0.1 tests and start a new folder for version 0.2 tests to support users who have not yet upgraded. Philippe Ombredanne expressed disagreement with this approach, arguing that maintaining parallel folders is confusing and that the group should instead use repository tags to preserve the old schema data, then migrate the entire repository to the new schema.
- **Compliance and Migration Planning**: John Horan noted that the current test suite (version 0.1) does not comply with 1st Edition ECMA-427 and requires updates, while simultaneously working on the 0.2 schema. The group discussed that once version 0.2 is published, the version 0.1 data should be considered frozen and available via tags, as maintaining two versions in parallel creates excessive complexity.
- **Release Tagging and Naming**: The participants agreed to rename the current purl-spec repo release title to "v1.0.0" and planned to publish the version 0.1 and 0.2 test schemas. They clarified that although they might tag the repository before and after changes, they would move all test data to use the version 0.2 schema rather than keeping multiple versions in parallel, as keeping them side-by-side creates a confusing and inconsistent user experience.
- **Demonstration of Versioning Risks**: Philippe Ombredanne demonstrated using the repository browser to show how users could access old versions of the schema via tags, reinforcing the argument that creating parallel directory paths for different schema versions is unnecessary and confusing for users.
- **Introduction of Atomic Test Cases**: Michael Herzog introduced the concept of "atomic test cases," proposing a structure of one test case per file to solve issues related to test case identification and organization.
- **Dimensions of Test Case Categorization**: The group discussed organizing test cases across three dimensions: implicit (specification versus types), explicit (required versus recommended), and functional (parse, build, and validate).
- **Challenges in Test Case Naming**: The participants debated the naming conventions for atomic test files, considering whether to use descriptive names or unique identifiers like numerical sequences, and how paths such as `required/spec/build/` could improve the ability for users to locate relevant tests.
- **Atomic Test Case Structure**: Philippe Ombredanne and Michael Herzog explored the benefits of having a clear file structure, noting that using a file-per-test approach ensures that file systems enforce uniqueness. John Horan questioned whether this approach might be difficult to navigate, but Philippe Ombredanne suggested that unique identifiers or simple numbering could suffice.
- **Prioritizing User Needs**: Michael Herzog and Philippe Ombredanne discussed the trade-off between easing the burden for the test maintainer versus the library implementer, reaching a consensus that the priority should be the implementer's experience while ensuring the structure is not unduly burdensome for maintainers.
- **Tooling for Test Maintenance**: The participants agreed that regardless of the file structure, they would likely need to develop scripts or a command-line utility to help maintainers aggregate or review tests, especially for identifying which specific test case fails in a given run.
- **Consensus on Folder Structure**: Philippe Ombredanne and Michael Herzog agreed on a proposed structure where the top-level folders would be categorized by "required" or "recommended" status, followed by "spec" or "types" segment, providing four clear paths for implementers to navigate.  See https://github.com/package-url/purl-spec/issues/765.
- **Next Steps for Atomic Testing**: Michael Herzog and John Horan committed to discussing the proposal further and finalizing the structure, including determining the necessary tools to support this shift.
- **Meeting Conclusion**: The participants acknowledged the progress made during the discussion and moved to close the meeting.
- The meeting was adjourned.
